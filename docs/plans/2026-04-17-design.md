# azex-ai/ledger — Design Document

> Production-grade double-entry ledger engine for Go.

**Created**: 2026-04-17
**Status**: Design approved, pending implementation
**Repo**: `github.com/azex-ai/ledger`

---

## 1. Project Overview

### Problem

Go 生态缺少一个生产级的复式记账库。现有方案要么是 ERP 级别的重量级系统，要么是玩具级 demo。开发者在金融、SaaS、游戏等场景反复手写 journal + checkpoint + 对账逻辑，容易出并发 bug。

### Solution

面向 Go 后端的通用复式记账引擎，提供两种使用方式：

1. **作为 Go 库** — `import "github.com/azex-ai/ledger/core"` 嵌入自己的服务
2. **作为独立服务** — HTTP API + Next.js 管理后台，Docker 一键部署

### Core Capabilities

- 复式记账（Journal + Entry，借贷平衡强校验）
- 动态科目体系（Classification / JournalType 运行时增删，支持弃用）
- 记账模板（EntryTemplate，预定义借贷配方，调用时填参数）
- 余额计算（Checkpoint + Delta 增量模式）
- Reserve / Settle / Lock（悲观锁 + 原子预留）
- Deposit / Withdraw 状态机（渠道抽象 + 链上示例实现）
- 对账引擎（会计恒等式校验、跨系统对账）
- 日快照 + 系统余额汇总
- Rollup Worker（异步 checkpoint 刷新）

### Tech Stack

- Go (chi v5 + pgx v5 + sqlc)
- PostgreSQL
- Next.js 16 + shadcn/ui + Tailwind v4

## 2. Architecture

### Package Structure

```
ledger/
├── core/                   ← 纯领域层（零外部依赖）
│   ├── account.go          ← Classification, AccountHolder, Currency 类型
│   ├── journal.go          ← Journal, Entry, EntryTemplate 类型 + 平衡校验
│   ├── checkpoint.go       ← BalanceCheckpoint, Rollup 类型
│   ├── reserve.go          ← Reserve/Settle/Lock 类型
│   ├── deposit.go          ← Deposit 状态机类型
│   ├── withdraw.go         ← Withdraw 状态机类型
│   ├── interfaces.go       ← 所有 Store 接口定义
│   ├── logger.go           ← Logger 接口 + nopLogger
│   └── metrics.go          ← Metrics 接口 + nopMetrics
├── postgres/               ← pgx + sqlc 适配器
│   ├── sql/                ← .sql 文件 + sqlc 生成
│   ├── migrations/         ← schema migrations (embed.FS)
│   ├── store.go            ← 实现 core 接口
│   └── migrate.go          ← schema migration
├── service/                ← 业务编排层（Rollup Worker, Reconciliation, Snapshot）
├── server/                 ← 独立 HTTP 服务（chi router + API）
├── web/                    ← Next.js 管理后台
├── examples/
│   └── crypto-deposit/     ← EVM CREATE2 链上充提完整 demo
└── cmd/
    └── ledgerd/            ← main.go 入口
```

### Account Dimensions

固定三维：`(AccountHolder int64, CurrencyID int64, ClassificationID int64)`

- AccountHolder 正数 = 用户/workspace，负数 = 系统对手方（`-userID`）
- 无需预创建账户，首笔 entry 自动建立
- 额外维度（如 ProjectID）可编码进 AccountHolder，聚合在应用层做

### Storage Abstraction

`core/` 定义接口，`postgres/` 是默认实现（唯一官方维护）。

想用 MySQL 的人自己实现 core 接口。README 明确标注 "PostgreSQL is the only officially supported database"。

### Observability

三个注入点，全部可选（不注入则静默）：

```go
engine := core.NewEngine(
    core.WithLogger(slogAdapter),
    core.WithMetrics(promAdapter),
    core.WithTracer(otelTracer),
    core.WithStore(pgStore),
)
```

## 3. Dynamic Classification & Template System

### Dynamic Classifications

科目不再硬编码为 Go 常量，存在数据库中，运行时 CRUD：

- `classifications` 表：`id, code, name, normal_side(debit/credit), is_system, is_active`
- `journal_types` 表：`id, code, name, is_active`
- 弃用逻辑：`is_active = false` 后，PostJournal 拒绝使用该科目/类型，历史数据保留

### Entry Templates

模板定义"一种业务动作对应哪些借贷分录"：

- `entry_templates` 表：`id, code, name, journal_type_id, is_active`
- `entry_template_lines` 表：`template_id, classification_id, entry_type, holder_role(user/system), amount_key, sort_order`

调用方式：

```go
err := ledger.ExecuteTemplate(ctx, "deposit_confirm", ledger.Params{
    HolderID:   userID,
    CurrencyID: currencyID,
    Amounts: map[string]decimal.Decimal{
        "amount": depositAmount,
        "fee":    feeAmount,
    },
    IdempotencyKey: txHash,
})
```

引擎内部：加载模板 → 替换 holder_role 为真实 ID → 替换 amount_key 为真实金额 → 组装 JournalInput → PostJournal（借贷平衡校验）。

借贷平衡在 **执行时** 由 PostJournal 的 `total_debit = total_credit` 约束兜底。

## 5. Deposit / Withdraw State Machines

### Design Principles

- `core/` 定义渠道接口 `ChannelAdapter` + 状态机，不含通信逻辑
- `examples/crypto-deposit/` 提供 EVM CREATE2 链上充提完整示例
- 回调幂等：`channel_ref UNIQUE` 防重复记账
- Confirm 使用实际到账金额，差额走 Suspense 调整

### Deposit State Machine（外 → 内）

```
pending → confirming → confirmed
                     → failed
                     → expired
```

| 状态 | 触发 | Journal |
|------|------|---------|
| `pending` | InitDeposit | DR Suspense(system) / CR Pending(user) |
| `confirming` | 渠道通知，等最终确认 | 无 |
| `confirmed` | ConfirmDeposit（实际金额） | 清 Pending/Suspense → DR MainWallet(user) / CR Custodial(system)。差额走 Suspense 调整 |
| `failed` | 渠道拒绝 | 反向冲正 Pending/Suspense |
| `expired` | 超时未确认 | 同 failed |

### Withdraw State Machine（内 → 外）

```
locked → reserved → reviewing → processing → confirmed
                  ↗                         → failed → reserved (retry)
                  └─ (auto skip if small) ──┘        → expired
```

| 状态 | 触发 | Journal |
|------|------|---------|
| `locked` | InitWithdraw | 创建提现意图，无记账 |
| `reserved` | ReserveWithdraw | DR MainWallet / CR Locked（Reserve 锁余额） |
| `reviewing` | 进入人工/规则审核（可配置跳过） | 无 |
| `processing` | 审批通过，提交渠道执行 | 无 |
| `confirmed` | 渠道确认完成 | DR Locked / CR Custodial（出金） |
| `failed` | 渠道拒绝 | 释放 Reserve，DR Locked / CR MainWallet |
| `expired` | 超时 | 同 failed（但不可 retry，reservation 已释放） |

**Retry**：`failed → reserved`（复用已有记录，不重复创建 reservation）。`expired` 不可 retry。

**Review 跳过**：小额提现可配置 `review_threshold`，低于阈值自动 `reserved → processing`。

### Concurrency

- Reserve 用 `pg_advisory_xact_lock(account_holder)` — 同一用户串行，不同用户并行
- Rollup 用 `SELECT ... WHERE processed_at IS NULL FOR UPDATE SKIP LOCKED` 消费
- Rollup 入队用 `INSERT ... ON CONFLICT DO NOTHING` 防并发 unique violation

### Partition Strategy

`journal_entries` 按 `created_at` RANGE 分区。`balance_checkpoints` 存 `last_entry_at` 用于分区裁剪，rollup 查询可限定时间范围避免跨分区扫描。

### Migration

内嵌 `embed.FS`，提供 `postgres.Migrate(ctx, pool)` 一键建表。兼容 golang-migrate 格式方便用户自管理。

## 9. Deployment & Open Source

### Deployment Modes

```yaml
# 1. 作为 Go 库嵌入（最轻）
go get github.com/azex-ai/ledger@latest

# 2. 独立后端服务（无前端）
docker run -e DATABASE_URL=postgres://... azex-ai/ledger:latest serve

# 3. 全功能（后端 + 前端 + 数据库）
docker compose up
```

### Docker Compose

- `ledgerd`：Go 后端（HTTP API + Rollup Worker + Reconciliation Cron）
- `web`：Next.js 管理后台
- `postgres`：带 migration 自动执行

### Configuration

环境变量优先，YAML 配置文件可选：

- `DATABASE_URL`
- `HTTP_PORT`
- `ROLLUP_INTERVAL`
- `ROLLUP_BATCH_SIZE`
- `RESERVATION_EXPIRE_DURATION`
- `RECONCILE_CRON`
- `SNAPSHOT_CRON`
- `WITHDRAW_REVIEW_THRESHOLD`
