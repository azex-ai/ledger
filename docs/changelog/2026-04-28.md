# Changelog - 2026-04-28

## [17:29] Transaction composition API — RunInTx + WithDB on every store

**类型**：feat(lib)
**概述**：Library 消费方现在可以把自己的业务写操作与 ledger 的 journal/booking 写操作放进同一个 PostgreSQL 事务，要么全部提交、要么全部回滚。这是构建「订单创建 + 资金锁定」等原子业务流的基础设施。

**详情**：

### postgres.DBTX 接口

新增 `postgres.DBTX` 接口（`sqlcgen.DBTX` + `Begin`）。`*pgxpool.Pool` 和 `pgx.Tx` 均满足该接口，所有 store 改为持有 `DBTX` 而非 `*pgxpool.Pool`。

### 每个 store 新增 `WithDB(db DBTX) *Store`

返回一个绑定到指定 `db`（pool 或 tx）的浅拷贝。原 store 不受影响。
原来在 pool 模式下需要自己开事务的两个路径（`GetBalance` 的 `REPEATABLE READ` 子事务、`ExecuteTemplateBatch` 的原子性）会在检测到外部 tx 时复用调用方事务。

### `ledger.Service.RunInTx(ctx, fn func(*Service) error) error`

```go
err = svc.RunInTx(ctx, func(tx *ledger.Service) error {
    _, err := tx.JournalWriter().PostJournal(ctx, input)
    return err
})
```

- fn 收到的 `*Service` 是短暂克隆（所有 store 指向同一事务）；fn 返回后即失效。
- fn 返回 nil → commit；非 nil error 或 panic → rollback。
- 默认隔离级别：READ COMMITTED。需要其他隔离级别的调用方应直接使用 `Pool().BeginTx`。
- advisory lock 在 tx 内获取会持到 commit/rollback（正确的余额锁语义）。
