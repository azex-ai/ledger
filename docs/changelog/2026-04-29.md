### 冲 (Deposit) — pending 两阶段 API

**分支**：`agent-delta-pending`

新增 `core.PendingBalanceWriter` 接口和 `postgres.PendingStore` 实现，将「冲账」分为两个原子步骤：

1. **AddPending** — DR suspense (system) / CR pending (user)：资金进入平台但尚未到账
2. **ConfirmPending** — DR pending (user) + DR main_wallet (user) / CR suspense (system) + CR custodial (system)：确认到账，清空 pending
3. **CancelPending** — DR pending (user) / CR suspense (system)：取消补偿分录（原始分录不变）

`core.PendingTimeoutSweeper` 接口：`ExpirePendingOlderThan(ctx, threshold)` 批量取消超时的 pending 余额。

门面方法：
- `svc.PendingBalanceWriter() core.PendingBalanceWriter`
- `svc.PendingTimeoutSweeper() core.PendingTimeoutSweeper`

Bundle 安装：`presets.InstallPendingBundle(ctx, ...)` — 安装 pending 分类和日志类型。

### 提 (Withdrawal) / 费 (Fee) — 扩展 presets 目录

**分支**：`agent-iota-observ`（presets 由 `3db12fe` / `19a121d` 引入）

`presets.InstallExtendedPresets` 现在安装 8 个 bundle：

| Bundle | 关键科目 | 关键日志类型 |
|--------|----------|------------|
| Deposit | pending, suspense, custodial | deposit_pending, deposit_confirm, … |
| Withdrawal | locked, fee_expense, fee_revenue | lock_funds, withdraw_confirm, withdraw_fee |
| Transfer | — | transfer_debit, transfer_credit |
| **Fee** | fees (system credit) | fee_charge |
| Capital | equity | capital_injection, capital_withdrawal |
| Settlement | settlement | checkout_settlement |
| Card | card_topup_pending | card_topup |
| Spread | spread_revenue | spread |

`fees` 科目是第一类费用汇总账户（等价于 payments 的 `AccountClassificationFees`），统一归集提现费、卡手续费、checkout 费等所有平台收入。

门面快捷方法：`svc.InstallExtendedPresets(ctx)` / `svc.InstallDefaultPresets(ctx)`

### 安全 (Security) — 余额趋势、平台余额、偿付能力检查

**分支**：`agent-epsilon-rollup`

#### 平台余额读取
`core.PlatformBalanceReader` — 从 `system_rollups` 物化视图按科目分类读取全平台余额：
- `GetPlatformBalances(ctx, currencyID)` → `*PlatformBalance{UserSide, SystemSide map[string]Decimal}`
- `GetTotalLiabilityByAsset(ctx, currencyID)` → 所有用户侧余额合计

`core.SolvencyChecker.SolvencyCheck(ctx, currencyID)` → `*SolvencyReport{Liability, Custodial, Solvent, Margin}`：
- `Custodial >= Liability` 时 `Solvent = true`
- O(1) 读取，直接查 `system_rollups` 物化表

门面方法：
- `svc.PlatformBalanceReader() core.PlatformBalanceReader`
- `svc.SolvencyChecker() core.SolvencyChecker`

#### 全量协调（10-check 套件）

**分支**：`agent-beta-reconcile`

`core.FullReconciler.RunFullReconciliation(ctx)` → `*ReconcileReport`，运行 10 项独立检查，任一失败不中止其余：

| # | 检查名 | 内容 |
|---|--------|------|
| 1 | `journal_dr_cr` | 所有分录借贷全局相等 |
| 2 | `checkpoint_balance` | Checkpoint 物化完整性（full-fleet scan 提示按账户运行） |
| 3 | `orphan_entries` | entries 引用不存在的 journal |
| 4 | `accounting_equation` | A = L + E，按 NormalSide 分类汇总 |
| 5 | `settlement_netting` | settlement 科目在宽限窗口外净余为零 |
| 6 | `non_negative_balances` | 用户账户余额 ≥ 0 |
| 7 | `orphan_reservations` | reservations.journal_id 引用不存在 |
| 8 | `pending_journal_timeout` | 预留（待 journals.status 字段迁移后启用） |
| 9 | `idempotency_uniqueness` | journals.idempotency_key 唯一性审计 |
| 10 | `stale_rollup_queue` | rollup_queue 中已过期 claim lease |

`service.FullReconciliationConfig` 提供可调参数：`EquationTolerance`、`SettlementClassCode`、`SettlementWindow`、`StaleRollupThreshold`、`NegativeBalancePageLimit`。

门面方法：`svc.FullReconciler(cfg) core.FullReconciler`

#### advisory lock 主节点选举

Worker 新增 `SetPool(*pgxpool.Pool)` — 为 `reconcile` 和 `system_rollup` 两个定时任务启用 `pg_try_advisory_lock` 主节点选举，多副本部署下每个 tick 只有一个 Pod 执行，单副本时无 lock 开销。

### 安全 (Security) — 快照稀疏存储 + 启动回填

**分支**：`agent-alpha-readapi`（snapshot 部分）

`core.SnapshotBackfiller.BackfillSnapshots(ctx, fromDate, toDate)` → `*BackfillResult`：
- 稀疏写入：仅在余额相对前一日快照发生变化时插入，节省存储
- `(*service.SnapshotBackfillService).CheckAndBackfillOnStartup` — 启动时检测快照缺口并自动回填
- 内置 advisory lock 防多副本重复回填

门面方法：`svc.SnapshotBackfiller() core.SnapshotBackfiller`

### 审计 (Audit) — 余额趋势 + 审计列表 + 全链路追踪

**分支**：`agent-alpha-readapi`

`core.BalanceTrendReader.GetBalanceTrends(ctx, filter)` → `[]BalanceTrendPoint`：
- 每个日历日返回一个点（Balance, Inflow, Outflow）
- 无活动的日期从上一已知余额前向填充
- 今日余额始终用 checkpoint+delta 实时值覆盖

`core.AuditQuerier` — 只读审计查询集合：

| 方法 | 用途 |
|------|------|
| `ListJournalsByAccount(ctx, filter)` | 按 (holder, currency, classification) 查询分录关联的 journal |
| `ListEntriesByJournal(ctx, journalID)` | 单 journal 所有条目 |
| `ListJournalsByTimeRange(ctx, filter)` | 时间范围查询 |
| `TraceBooking(ctx, bookingID)` | 返回 booking + 全部 events + 全部 journals |
| `ListReversals(ctx, journalID)` | 返回 journal 的完整冲销链 |

门面方法：
- `svc.BalanceTrends() core.BalanceTrendReader`
- `svc.Audit() core.AuditQuerier`

### 审计 (Audit) — OTEL trace 传播 + events 新字段

**分支**：`agent-iota-observ`

`core.Event` 新增两个字段：
- `ActorID int64` — 触发此次 transition 的用户/系统 actor（0 = 系统）
- `Source string` — 调用来源（如 `"api"`, `"worker"`, `"webhook"`）

这两个字段透传进 `TransitionInput`，写入事件行，方便支持查询「谁在什么渠道触发了这次状态变更」。

OTEL 追踪：`core.Engine` 已支持从 `context.Context` 提取 OpenTelemetry span，journal/booking 写操作自动记录 span attributes（journal_type, template_code, booking_id, to_status）。与现有 Prometheus metrics 并行使用；NopMetrics/NopLogger 保持为默认值，零依赖。
